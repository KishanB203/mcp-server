## Project Context
This project is a multi-agent AI DevOps pipeline that automates:
Azure DevOps → Figma Design → Clean Architecture → GitHub PR → Code Review → Merge

## Rules (MUST FOLLOW)
Before generating any code, Claude MUST read and follow:
- `rules/coding-standard.md` — code style, naming, error handling
- `rules/architecture.md` — clean architecture layers
- `rules/naming-rule.md` — file, variable, function naming
- `rules/testing-rule.md` — test structure and coverage requirements

## Available MCP Tools

### Azure DevOps
- `ado_get_work_item(id)` — Fetch task details
- `ado_list_work_items(filters)` — List/filter tasks
- `ado_update_work_item_state(id, state)` — Update task state
- `ado_add_comment(id, comment)` — Add comment to task

### Figma Design
- `figma_get_file()` — Check Figma connection
- `figma_generate_wireframe(taskId)` — Generate wireframe spec + link to ADO
- `figma_add_ado_link(adoTaskId, figmaUrl)` — Add Figma link to ADO task

### Branch & PR
- `branch_generate_name(taskId, taskTitle)` — Generate `feature/{id}-{name}`
- `conflict_preflight_check(taskId, branchName)` — Check for duplicate branches/PRs

### Agents
- `agent_analyze_task(taskId)` — ProductOwnerAgent: analyze task completeness
- `agent_generate_architecture(featureName)` — ArchitectAgent: scaffold clean architecture
- `agent_start_development(taskId)` — DeveloperAgent: create branch, mark In Progress
- `agent_create_pr(taskId, branchName)` — DeveloperAgent: create GitHub PR
- `agent_review_pr(prNumber, branchName)` — ReviewerAgent: automated code review
- `agent_merge_pr(prNumber)` — DevOpsAgent: merge PR, close ADO task
- `agent_full_workflow(taskId)` — **Run the entire pipeline in one call**

## Standard Workflow: "Work on Task 123"

When the user says "work on task 123", "implement task 456", or similar:

1. **Call `agent_full_workflow(taskId)`** — this handles everything automatically:
   - Fetches the ADO task
   - Analyzes it (ProductOwnerAgent)
   - Generates Figma wireframe (if UI task)
   - Scaffolds clean architecture
   - Creates `feature/{id}-{name}` branch
   - Marks task as "In Progress"
   - Creates a PR with the standard template

2. **After scaffold is created**, implement the actual business logic:
   - Fill in domain entities in `src/domain/{feature}/`
   - Implement use cases in `src/application/{feature}/use-cases/`
   - Wire up infrastructure in `src/infrastructure/{feature}/`
   - Build UI in `src/ui/{feature}/`

3. **Then commit and review**:
   - `agent_create_pr(taskId, branchName)`
   - `agent_review_pr(prNumber, branchName)`
   - `agent_merge_pr(prNumber)`

## Branch Naming Convention
```
feature/{task-id}-{kebab-case-title}
```
Examples:
- `feature/123-add-employee-form`
- `feature/456-fix-login-redirect`

**Never** create a branch manually — always use `branch_generate_name`.

## PR Title Format
```
[Task-ID] Task Title
```
Example: `[123] Add employee management form`

## Commit Message Format
```
feat: {title} [#{id}]
```
Example: `feat: Add employee management form [#123]`
